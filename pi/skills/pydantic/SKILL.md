Use these rules whenever writing or reviewing Python Pydantic schemas.

- Treat SQLAlchemy models and column metadata as the primary source of truth for field semantics.
- Prefer readability, maintainability, and speed over convenience wrappers.
- Target the repository’s pinned/latest Pydantic version in use.

## 1) Core typing and schema parity with SQLAlchemy

### 1.1 Match DB typing

For each schema field:

- Use the SQLAlchemy type family as the baseline for Pydantic type choices.
- Preserve nullability semantics:
  - if the DB column is nullable, the Pydantic field should be optional (or equivalent nullable typing);
  - if non-nullable, make it required unless a business rule is explicitly documented as an exception.
- Preserve enum/string/number semantics and precision expectations from the DB contract.

### 1.2 Optionality exceptions

Business-logic exceptions are allowed only when necessary and should be explicitly documented at the schema level (docstring/field metadata/comment).

- Do **not** change required/optional behavior for convenience.
- Do **not** widen a non-nullable DB field into optional in create/update request/response schemas unless required by an explicit business rule.

### 1.3 Legacy naming and legacy DB conventions

- Prefer `snake_case`, lower-case attributes.
- Prefer singular naming for entities and fields where practical.
- When legacy or pluralized DB naming is required, keep model semantics clean via explicit mapping (field alias / config mapping).
- Keep mapping explicit rather than embedding ambiguity in multiple field copies.

## 2) Schema architecture: base models, composition, and inheritance

### 2.1 Prefer reusable base categories

Use category-based base models to reduce duplication:

- `*Base` – scalar fields common across contexts.
- `*Create` – create-only fields and constraints.
- `*Read` – read/response-only fields.
- `*WithRelations` – internal relation payloads reused in multiple schemas.
- `*RelationsExternal` – cross-boundary relation representations.
- `*RelationsGenerated` – fields generated by system behavior (timestamps, computed IDs, etc.).

Compose final schemas from these categories via inheritance.

### 2.2 Merge category models intentionally

- Combine model categories into a final class when it reduces duplication and keeps intent clear.
- Avoid over-composition that makes MRO/field resolution difficult to reason about.
- Group by domain lifecycle: **read/create/update/delete/response** should reuse a shared base and diverge only where needed.

### 2.3 Scope of base reuse

- Reuse base models across related schemas by default.
- Reusing base models across unrelated domain models is advisory only; prefer local, scoped models when structure/noise suggests limited shared semantics.

## 3) Discriminated unions for form state schemas

Use discriminated unions for action-driven forms instead of manual branching logic.

### 3.1 Required action contract

- Canonical discriminator values: `create`, `update`, `delete`, `none`.
- Include:
  - `action: Literal["create"|"update"|"delete"|"none"]`
- Define separate schemas per action and merge via a discriminated union.

### 3.2 Create-state identifier handling

- `create` schemas must use temporary link identifiers (typically UUID-style `_id` fields) for records that will later be persisted to sequence-based DB PKs.
- Non-`create` states should use canonical DB-facing identifiers.
- Define this transformation boundary clearly in model docs and naming.

### 3.3 State schema organization

- Build `Create`, `Update`, `Delete`, and `None` state schemas via shared bases to avoid duplicated field definitions.
- Keep discriminator-specific validation strict.

## 4) FastAPI response performance and design pattern

- Always return typed Pydantic response models from FastAPI handlers.
- Do not hand-iterate and manually hydrate response objects field-by-field when data already fits schema; rely on FastAPI/Pydantic serialization.
- Keep response path fast: transform only at data boundaries where needed.

## 5) Object modeling vs `Dict`

- Do **not** use `Dict` as a shortcut for structured objects.
- Use explicit `BaseModel` hierarchies for nested and relational payloads.
- Use `Dict` only for truly dynamic, unconstrained key/value domains, and only with explicit justification.

## 6) Validation policy

- Prefer fail-fast validation.
- Keep transforms minimal.
- Use transforms/normalization only when business rules demand it (for example: strict currency precision/decimal-place constraints).
- Otherwise, let invalid inputs fail early at model boundary.

## 7) Relationship modeling

- Model relationships with explicit relation models (and relation enums/typed refs), not raw key-value maps.
- Use composable relation model slices (`*WithRelations`, `*RelationsExternal`) and include them only where used.

## 8) Prohibited patterns (avoid)

- Missing discriminator for action-state schemas where branch-specific validation is needed.
- Generic dict/object payloads replacing typed related models.
- Implicit drift from SQLAlchemy types or nullability.
- Overly broad inheritance that obscures required fields.
- Manual coercive response mapping loops that duplicate validation and serialization work.

## 9) Authoring checklist

Before finalizing a schema change, verify:

1. SQLAlchemy parity: field types and optionality match DB intent.
2. Naming: `snake_case`, lower-case, and explicit aliasing for legacy/legacy-plural DB names.
3. Composition: relevant `Base/Create/Read/Relations*` classes are used to reduce duplication.
4. Action schemas: discriminated union is defined for `create|update|delete|none` and includes all states.
5. Create behavior: required `_id`-style temp identifier policy exists where create links to DB-generated IDs.
6. Relation modeling: relational payloads are explicit models, not dict placeholders.
7. Validation strategy: mostly fail-fast, transforms only for justified exception cases.
8. FastAPI responses: returning typed Pydantic models directly, without manual per-field hydration.
9. Reuse decisions: unrelated base reuse has clear justification.
